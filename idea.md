# local-rag — когнитивный слой для AI агентов

## Проблема

Современные AI агенты (Claude Code и подобные) лишены памяти между сессиями.
Каждая новая сессия начинается с нуля — без знания о том, что уже решено,
что находится в работе, какие вопросы остались открытыми.

Пользователь вынужден каждый раз восстанавливать контекст вручную.
Агент вынужден каждый раз строить картину проекта заново — переспрашивая,
переосмысливая, иногда повторно решая уже решённые задачи.

Это не проблема интерфейса. Это архитектурная проблема.

## Идея

local-rag — это не инструмент для пользователя.
Это когнитивная инфраструктура для AI агента.

Система строит и поддерживает семантическую память проекта:
что известно, что в работе, что решено, что осталось открытым вопросом.
Эта память живёт локально, в векторной базе Qdrant, и обновляется автоматически
по ходу каждой сессии — без участия человека.

Агент получает контекст при старте сессии и обогащает память по её завершении.
Человек ставит задачу и принимает результат. Всё остальное происходит между агентом и памятью.

## Ключевые принципы

**Память — не архив, а живое состояние.**
Каждая запись имеет статус: в работе, решено, открытый вопрос, гипотеза.
Статус меняется по ходу работы. Старые решения не удаляются — они помечаются как resolved.

**Неизвестность — тоже результат.**
Если поиск в памяти ничего не нашёл — это не ошибка. Это сигнал: территория новая,
всё что будет открыто в этой сессии должно быть запомнено.

**Агент — валидатор, не пассивный потребитель.**
Лёгкая локальная модель (Gemma4 через Ollama) анализирует транскрипт сессии
и предлагает операции с памятью. Агент подтверждает или корректирует через MCP.
Решение о том что запомнить — не алгоритмическое и не человеческое. Это совместное.

**Приватность по умолчанию.**
Код, файлы, контекст проекта не покидают машину пользователя.
Облачные LLM провайдеры используются только для генерации эмбеддингов и роутинга —
без передачи содержимого кодовой базы.

**Минимальный порог входа.**
Пользователю нужно: Qdrant (Docker) + LLM провайдер (Ollama локально или API ключ).
Две команды: `npx @13w/local-rag init` и `npx @13w/local-rag serve`.

## Чем это отличается от других RAG систем

Большинство RAG систем строятся для людей — как поиск по документам,
как чат с PDF, как умный поиск по кодовой базе.

local-rag строится для агента. Человек не в контуре памяти.

Разница принципиальная:
- Обычный RAG: пользователь задаёт вопрос → система ищет → пользователь читает ответ
- local-rag: агент начинает сессию → получает контекст → работает → память обновляется → следующая сессия умнее

Это не поисковый инструмент. Это долговременная рабочая память агента.

## Почему это не закроет Anthropic

Anthropic строит облачную память: факты о пользователе, история разговоров, общий контекст.
Это полезно, но принципиально другое.

local-rag хранит то, что Anthropic никогда не потащит в облако:
локальный код, приватные файлы, контекст конкретного проекта.

Семантический поиск по приватной кодовой базе — это не функция которую можно
добавить в облачный сервис без нарушения приватности пользователя.
Именно поэтому это остаётся локальным — и именно поэтому это нужно.

## Для кого

Прежде всего — для разработчиков использующих Claude Code на больших проектах.
Для команд где приватность кода является требованием, а не предпочтением.
В перспективе — для любого AI агента которому нужна персистентная рабочая память.
